# No Attention, No Problem: DPU-Aware Attention Approximation in Modern YOLO on FPGA

**论文**: [arXiv](https://arxiv.org/abs/2607.13106)  
**任务**: FPGA 上的实时 YOLO 部署

## 一句话总结

论文不尝试让 FPGA DPU 强行执行现代 YOLO 的注意力算子，而是从编译器支持和内存布局出发，把不支持的空间注意力近似为 DPU 友好的卷积结构，并通过 row-stacking 等重排方式减少张量变换开销，在 INT8 部署中联合比较精度、吞吐、延迟与功耗。

## 背景与问题

YOLO11/12/26 一类模型逐渐加入 PSA、矩阵乘、动态 reshape 和归一化操作，但商用 FPGA DPU 通常只高效支持固定卷积、池化、激活与少量张量算子。直接导出会出现算子落到 CPU、图被切分或完全无法编译；理论上更轻的注意力模型反而可能比纯卷积版本更慢。

## 方法总览

作者先分析目标 DPU 的算子白名单和张量布局限制，再逐项定位现代 YOLO 中无法编译或会触发 CPU fallback 的注意力计算。随后以静态卷积、池化和逐元素运算近似空间调制，并重排输入布局减少 transpose，最终对替换后的网络执行 INT8 量化和板端编译。

## 方法详解

```mermaid
flowchart LR
    M[现代 YOLO 图] --> U[定位 DPU 不支持算子]
    U --> A[注意力的卷积近似]
    U --> R[Row-stacking 数据重排]
    A --> Q[INT8 量化与编译]
    R --> Q
    Q --> F[FPGA DPU]
    F --> E[精度/延迟/功耗评估]
```

### DPU-aware 近似

作者保留注意力“根据空间上下文调制特征”的功能目标，但将矩阵注意力替换为由卷积、池化和逐元素运算组成的静态图。近似模块的价值不在 FLOPs 更低，而在于全部算子可以留在 DPU 上，避免主机与加速器之间频繁切换。

### Row-stacking

某些二维扫描或重排会产生低效 transpose。Row-stacking 将行方向数据预先堆叠成适合 DPU 连续访问的布局，用卷积实现原本的邻域交互，降低内存搬运和图切分。

### 量化与系统评价

方法围绕 INT8 编译设计，并将板端预处理、DPU 执行和后处理纳入端到端测量。论文强调 FPGA 方案不能只报告模型 FPS，还要同时报告功耗与精度损失。

## 实验与证据

- 比较原始注意力 YOLO、去注意力版本与 DPU-aware 近似版本。
- 报告浮点模型、量化模型和 FPGA 实机结果，区分结构替换损失与 INT8 损失。
- 结果说明完全删除注意力虽然易部署，但精度下降更明显；近似结构能在 DPU 可执行前提下保留部分收益。
- 系统结果同时覆盖吞吐、单帧延迟与能效，避免只用桌面 GPU 指标推断 FPGA 表现。

## 对 YOLO-Agent 的启发

- 为目标后端建立算子白名单，Harness 在生成架构前就拒绝不可编译组件。
- 将“原模块、直接删除、DPU 近似”设为三联对照，并保存量化前后逐层误差。
- 指标必须包括编译成功率、CPU fallback 比例、板端功耗和完整管线延迟。
- 对近似模块做多尺度特征可视化，检查是否损害小目标和细粒度边缘。

## 优点

- 从目标编译器和硬件约束反向设计网络，而不是只追求理论 FLOPs。
- 区分结构替换损失、量化损失和系统调度损失，便于定位部署问题。
- 同时报告精度、延迟与功耗，适合边缘设备的真实决策。

## 局限

- 方案高度依赖特定 DPU 编译器和支持算子集合。
- 近似注意力不等价于原始全局关系建模，模型规模增大后差距可能扩大。
- FPGA 上的优势不能直接推广到 GPU、NPU 或移动端 DSP。
